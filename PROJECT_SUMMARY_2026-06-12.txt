DESCRIPTION

Complete Infrastructure as Code (IaC) project to deploy a full AWS infrastructure
using Terraform. This is a production-ready example with best practices for
managing multi-environment deployments (dev/staging/prod).

✓ AWS Security Groups & VPC Networking
✓ Multi-environment management with tfvars files
✓ Makefile for automation

================================================================================
ARCHITECTURE & RESOURCES DEPLOYED
================================================================================

VPC Networking:
  - VPC with CIDR 10.0.0.0/16
  - 2 Public Subnets (for ALB and NAT Gateway)
  - 2 Private Subnets (for EC2 and RDS)
  - Internet Gateway for public internet access
  - 2 NAT Gateways for private subnet egress
  - Route tables (public + 2 private per AZ)

Security:
  - 3 Security Groups (ALB, App EC2, RDS PostgreSQL)
  - Ingress rules by use case
  - Egress rules allowing all outbound

Compute & Database:
  - 2 EC2 Instances (t3.micro) in private subnets with Auto Scaling potential
  - RDS PostgreSQL (Multi-AZ, encrypted, automated backups)
  - CloudWatch Log Group for RDS PostgreSQL logs
  - User data script to bootstrap web servers

Load Balancing:
  - Application Load Balancer (ALB) on public subnets
  - Target Group with health checks
  - HTTP routing to EC2 instances

HOW TO USE
================================================================================

1. Initialize Terraform:
   $ cd projects/2026-06-12_terraform-aws-infrastructure
   $ terraform init

2. Validate configuration:
   $ terraform validate
   $ terraform fmt -check -recursive

3. Plan the infrastructure:
   $ terraform plan -var-file="environments/dev.tfvars"

4. Apply (Deploy):
   $ terraform apply -var-file="environments/dev.tfvars"

5. View outputs:
   $ terraform output
   $ terraform output -raw alb_dns_name

6. Cleanup (Destroy):
   $ terraform destroy -var-file="environments/dev.tfvars"

Or use the convenient Makefile:
   $ make plan ENVIRONMENT=dev
   $ make apply ENVIRONMENT=prod
   $ make destroy ENVIRONMENT=staging

NOTES
================================================================================

- The db_password variable is marked as sensitive to prevent exposure in logs
- All resources are tagged with Environment, Project, ManagedBy, and CreatedAt
- AWS credentials must be configured locally (~/.aws/credentials)
- The infrastructure creates resources that may incur AWS charges
- Always run 'terraform plan' before 'apply' to review changes
- For production use, implement remote state backend (S3 + DynamoDB)
- Never commit terraform.tfstate or secrets to git
- Use environment variables (TF_VAR_*) or .auto.tfvars for sensitive data
